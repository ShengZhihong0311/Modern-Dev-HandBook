## 6.0 概念基础：范式与实践

在进入 AI 原生开发之前，先理解两个核心概念："范式"和"实践"。这两个词在技术讨论中频繁出现，理解它们的含义，才能更好地把握技术演进和学习方向。

### 什么是"范式"（Paradigm）

范式是一套被广泛认可的思维方式、工作模式和方法论框架。它不是具体的技术或工具，而是解决问题的"标准思路"。

**范式的特征**：

| 特征 | 说明 |
|------|------|
| 隐性共识 | 不需要明文规定，团队成员默认遵循 |
| 标准思路 | 针对某类问题，有一套公认的解决路径 |
| 时代性 | 随技术演进而变化，旧范式会被新范式替代 |
| 认知框架 | 决定了"怎么思考问题"，而非"怎么操作" |

**范式示例**：

```
面向对象范式（OOP）
├── 核心思想：用"对象"组织代码，封装数据和行为
├── 标准思路：先设计类，再设计继承关系，最后实现细节
└── 代表语言：Java、C++、Python

函数式范式（FP）
├── 核心思想：函数是第一公民，避免状态变化
├── 标准思路：用纯函数组合，而非对象继承
└── 代表语言：Haskell、Clojure、Scala

AI 原生开发范式（Vibe Coding）
├── 核心思想：描述意图，AI 实现细节
├── 标准思路：先清晰表达目标，再审查 AI 输出
└── 代表工具：Claude Code、Cursor、Copilot
```

**什么时候说"全新范式"**？

当思维方式彻底改变，不是简单的工具升级或效率提升。范式转变意味着：你解决问题的思路变了，工作重心变了，核心技能变了。

| 变化类型 | 是否范式转变 | 示例 |
|----------|--------------|------|
| 工具升级 | 不是 | 从 Vim 到 VS Code |
| 效率提升 | 不是 | 从手动测试到自动化测试 |
| 思维方式改变 | 是 | 从"写代码"到"描述意图" |
| 工作重心转移 | 是 | 从"编码"到"审查" |

### 什么是"实践"（Practice）

实践是在特定场景下被验证有效的具体做法、流程、技巧。它不是抽象的理念，而是可执行的步骤。

**实践的特征**：

| 特征 | 说明 |
|------|------|
| 可执行 | 有明确的操作步骤，可以直接照做 |
| 可复制 | 不同人执行，能得到类似效果 |
| 有明确效果 | 实践后有可衡量的改进 |
| 经时间验证 | 经过多次验证，被广泛认可 |

**实践示例**：

```
单元测试实践
├── 具体做法：为每个函数写测试，覆盖正常+边界+异常
├── 效果：改动代码后快速发现问题
└── 验证：业界几十年验证，质量提升明显

Code Review 实践
├── 具体做法：PR 合并前至少一人审查代码
├── 效果：减少 bug、知识共享、规范统一
└── 验证：Google、Microsoft 等大厂长期使用

结构化提示词实践
├── 具体做法：背景 + 目标 + 约束 + 期望
├── 效果：AI 输出更贴合需求，减少迭代轮数
└── 验证：AI 原生开发者反复验证
```

**什么是"最佳实践"**？

同类实践中效果最好、最被认可的方案。"最佳"不是绝对最好，而是"当前被广泛认可的优选方案"。最佳实践会随时间演进，今天的最佳实践可能明天就被更好的方案替代。

| 说法 | 含义 |
|------|------|
| "这是最佳实践" | 业界广泛认可，效果验证，推荐使用 |
| "遵循最佳实践" | 采用被验证的有效做法，而非自己摸索 |
| "更新最佳实践" | 有更好的方案出现，需要更新认知 |

### 范式 vs 实践：层级对比

范式和实践是不同层级的概念：范式是宏观的思维框架，实践是微观的具体做法。

| 维度 | 范式 | 实践 |
|------|------|------|
| 层级 | 思维框架（宏观） | 具体做法（微观） |
| 稳定性 | 相对稳定，时代性变化 | 持续演进，场景性变化 |
| 学习方式 | 理解原理、转变认知 | 学习步骤、反复练习 |
| 关系 | 定义"怎么思考" | 在范式框架下"怎么操作" |
| 示例 | "面向对象编程范式" | "私有属性用下划线前缀" |

**理解这个区别的价值**：

1. **听技术讨论时**：能分辨对方在说"思维方式"还是"具体做法"
2. **学习新技术时**：先理解范式（为什么这样设计），再学实践（具体怎么操作）
3. **评估技术方案时**：范式决定方向是否正确，实践决定执行是否高效

### 本书的范式与实践

本书涉及的范式和实践分布：

| 章节 | 范式 | 实践 |
|------|------|------|
| 第一章 | 环境隔离范式 | uv 管理依赖、虚拟环境配置 |
| 第二章 | CLI 交互范式 | 管道组合、输出处理 |
| 第三章 | 问题诊断范式 | 日志分析、异常追踪 |
| 第四章 | 版本控制范式 | 分支策略、Commit 格式 |
| 第五章 | 容器化范式 | Dockerfile 编写、镜像优化 |
| 第六章 | AI 原生开发范式 | 提示词结构、代码审查 |
| 第七章 | 工程规范范式 | 测试覆盖、安全检查 |

## 6.1 AI 原生开发的本质

传统开发：开发者理解需求 → 设计方案 → 编写代码 → 调试修复。AI 原生开发：开发者描述意图 → AI 生成方案 → 人机协作迭代 → 快速交付。

这不是"让 AI 写代码"，而是"让 AI 理解你的意图"。核心转变：从语法层面的编码，升级到语义层面的描述。你不再是"翻译官"（把想法翻译成代码），而是"架构师"（描述目标，AI 执行细节）。

### 传统开发 vs AI 原生开发

| 维度 | 传统开发 | AI 原生开发 |
|------|----------|-------------|
| 核心技能 | 语法精通、API 记忆 | 意图描述、上下文组织 |
| 工作重心 | 写代码（60%）+ 调试（30%） | 设计（40%）+ 审查（40%）+ 调试（20%） |
| 学习曲线 | 每个框架都要学 | 理解原理，AI 补细节 |
| 错误来源 | 语法错误、API 误用 | 意图不清、上下文不足 |
| 效率瓶颈 | 编码速度 | 思考清晰度 |

### Vibe Coding 三层认知

```
┌─────────────────────────────────────┐
│  Layer 3: 认知层 — 知道要做什么      │  ← 你的核心价值
│  （需求理解、架构决策、技术选型）    │
├─────────────────────────────────────┤
│  Layer 2: 描述层 — 清晰表达意图      │  ← 你的新技能
│  （提示词设计、上下文组织）          │
├─────────────────────────────────────┤
│  Layer 1: 执行层 — AI 实现细节       │  ← AI 的强项
│  （语法生成、API 调用、代码填充）    │
└─────────────────────────────────────┘
```

## 6.2 与 AI 协作的工作流

AI 原生开发不是一次性"让 AI 写完"，而是人机协作迭代。核心流程：描述 → 生成 → 审查 → 修正 → 验证 → 迭代。

关键原则：不要指望 AI 一次完美；每次迭代都要提供反馈；审查代码比写代码更重要；验证结果比信任 AI 更可靠。

### 标准协作流程

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  描述意图 │ → │  AI 生成  │ → │  人工审查 │ → │  验证测试 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                                                    ↓
      └───────────────── 修正反馈 ←───────────────────────┘
```

### 实战工作流示例

```bash
# === 场景：添加用户认证功能 ===

# 1. 描述意图（清晰、具体、有上下文）
# "在 FastAPI 项目中添加 JWT 用户认证：
#  - 用户注册/登录接口
#  - 密码用 bcrypt 加密
#  - Token 有效期 24 小时
#  - 需要中间件保护 /api/* 路由
#  项目使用 uv 管理，已有 pyproject.toml"

# 2. AI 生成代码（阅读、理解、不要盲目复制）

# 3. 审查代码（检查三件事）：
#  - 安全性：密码处理是否正确？
#  - 兼容性：是否符合现有项目结构？
#  - 可维护性：代码是否清晰易懂？

# 4. 验证测试
uv run pytest tests/auth/             # 运行测试
curl -X POST http://localhost:8000/register  # 手动验证

# 5. 反馈修正（如有问题）
# "登录接口返回 500，原因是数据库连接未配置，
# 请修改为从环境变量读取 DATABASE_URL"
```

## 6.3 提示词工程：清晰表达意图

提示词质量决定 AI 输出质量。好的提示词 = 清晰的意图 + 充足的上下文 + 具体的约束。

核心原则：说清楚"要做什么"、"在什么环境下"、"有什么约束"、"期望什么输出"。模糊的提示词得到模糊的结果。

### 提示词结构模板

```
【背景】项目是什么？技术栈是什么？当前状态？
【目标】要实现什么功能？解决什么问题？
【约束】有什么限制？不能用什么？必须用什么？
【期望】代码风格？文件结构？输出格式？
```

### 好提示词 vs 坏提示词对比

| 维度 | 坏提示词 | 好提示词 |
|------|----------|----------|
| 具体性 | "写个登录功能" | "用 FastAPI + JWT 实现登录，密码 bcrypt 加密" |
| 上下文 | 无上下文 | "项目结构：src/api/，使用 uv 管理，已有数据库连接" |
| 约束 | 无约束 | "不要用 SQLAlchemy ORM，用原生 SQL" |
| 期望 | 无期望 | "代码放在 src/api/auth.py，包含单元测试" |

### 实战提示词示例

```markdown
# === 好的提示词示例 ===

【背景】
这是一个 FastAPI 项目，使用 uv 管理依赖。
项目结构：
- src/api/routes/（路由）
- src/models/（数据模型）
- src/services/（业务逻辑）
已有 PostgreSQL 数据库连接。

【目标】
添加用户认证模块，包括：
1. 用户注册接口 POST /api/register
2. 用户登录接口 POST /api/login，返回 JWT Token
3. 认证中间件，保护 /api/* 路径

【约束】
- 密码使用 bcrypt 加密，强度 12
- JWT 有效期 24 小时
- 需要处理用户名重复、密码错误等异常
- 不使用 ORM，用 asyncpg 原生 SQL

【期望】
- 代码放在 src/api/auth.py
- 包含类型注解
- 包含错误处理
- 提供 pytest 测试文件
```

## 6.4 上下文组织：让 AI 理解你的项目

AI 不知道你的项目结构、依赖关系、编码风格。你需要主动提供上下文。上下文越多，AI 输出越贴合你的项目。

核心方法：项目结构文件（CLAUDE.md）、关键代码片段、依赖说明、编码规范。

### CLAUDE.md：项目的导航地图

CLAUDE.md 是给 AI 的项目说明书，让 AI 快速理解项目架构、关键约束、开发规范。

```markdown
# CLAUDE.md 示例

## 项目定位
乡村智慧决策系统，基于 LangChain/LangGraph 的微服务架构。

## 目录结构
├── service/          # FastAPI 后端
├── src/agents/       # Agent 系统
├── src/algorithms/   # 检测算法
├── frontend/         # Next.js 前端

## 核心约束
1. 必须使用 uv run 运行 Python
2. LangChain/LangGraph 语法问题需查阅官方文档
3. 每次修改后运行健康检查

## 常用命令
uv run python run_server.py    # 启动后端
bash scripts/dev/check.sh      # 健康检查

## 关键文件
- service/server.py: 主服务入口
- src/agents/orchestrator.py: Agent 编排
```

### 提供上下文的最佳时机

| 场景 | 应提供什么 |
|------|-----------|
| 新功能开发 | 项目结构、相关文件路径、依赖版本 |
| Bug 修复 | 错误日志、相关代码片段、复现步骤 |
| 代码重构 | 当前代码、目标结构、保留的功能 |
| API 集成 | API 文档、现有集成示例、期望的封装方式 |

## 6.5 AI 辅助代码审查

AI 生成的代码不一定正确、安全、高效。审查是开发者的核心责任。审查重点：安全性、正确性、可维护性、性能。

### 审查检查清单

| 类别 | 检查项 | 风险 |
|------|--------|------|
| 安全性 | 输入验证、权限检查、敏感数据处理 | 高危漏洞 |
| 正确性 | 逻辑正确、边界处理、异常处理 | 功能缺陷 |
| 可维护性 | 代码清晰、命名规范、注释充分 | 维护困难 |
| 性能 | 资源使用、算法效率、并发安全 | 性能瓶颈 |
| 兼容性 | 依赖版本、接口兼容、平台兼容 | 部署失败 |

### 实战审查示例

```python
# AI 生成的代码（需要审查）
def login(username: str, password: str):
    user = db.query(f"SELECT * FROM users WHERE username='{username}'")
    if user and user.password == password:
        return create_token(user.id)
    return None

# === 审查发现的问题 ===
# 1. 【高危】SQL 注入：username 直接拼接 SQL
# 2. 【高危】明文密码：password 直接比较
# 3. 【中危】无日志：登录失败无记录
# 4. 【低危】无类型：返回值类型不明确

# === 审查后的修正代码 ===
async def login(username: str, password: str) -> TokenResponse | None:
    # 使用参数化查询防止 SQL 注入
    user = await db.fetchrow(
        "SELECT id, password_hash FROM users WHERE username = $1",
        username
    )
    if user and verify_password(password, user["password_hash"]):
        logger.info(f"User {username} logged in")
        return create_token(user["id"])
    logger.warning(f"Login failed for {username}")
    return None
```

Cursor 是 VS Code 的 AI 强化版，深度集成 AI 能力。

```
# === Cursor 核心功能 ===

# Cmd+L：Chat 模式（项目级问答、代码解释）
```

## 6.7 常见陷阱与应对

AI 原生开发有独特陷阱：过度信任、上下文不足、意图不清、盲目复制。识别陷阱，才能避免。

### 陷阱分类与应对

| 陷阱 | 症状 | 根因 | 应对策略 |
|------|------|------|----------|
| 过度信任 | 代码有 bug、安全漏洞 | 未审查 AI 输出 | 每次生成必审查 |
| 上下文不足 | 生成代码与项目不兼容 | 未提供项目信息 | 先写 CLAUDE.md |
| 意图不清 | 生成结果偏离目标 | 提示词模糊 | 使用结构化模板 |
| 盲目复制 | 不理解代码逻辑 | 未阅读生成代码 | 先理解再使用 |
| 依赖幻觉 | 引用不存在库/函数 | AI 预训练知识过时 | 验证 API 存在性 |
| 安全忽视 | SQL 注入、明文密码 | 未强调安全约束 | 明确安全要求 |

### 安全审查重点

```python
# === AI 生成代码常见安全问题 ===

# 1. SQL 注入（最常见）
# 坏：f"SELECT * FROM users WHERE id={user_id}"
# 好："SELECT * FROM users WHERE id = ?", (user_id)

# 2. 命令注入
# 坏：os.system(f"ls {filename}")
# 好：subprocess.run(["ls", filename], check=True)

# 3. 路径遍历
# 坏：open(f"/data/{filename}")
# 好：open(os.path.join(BASE_DIR, secure_filename(filename)))

# 4. 敏感信息泄露
# 坏：return {"password": user.password}
# 好：return {"username": user.username}  # 不返回敏感字段

# 5. 弱加密
# 坏：hashlib.md5(password.encode())  # MD5 已不安全
# 好：bcrypt.hashpw(password.encode(), bcrypt.gensalt(12))
```

## 6.8 AI 原生开发的心态转变

成功驾驭 AI，需要心态转变：从"怀疑 AI"到"审慎信任"；从"写代码"到"设计系统"；从"记忆 API"到"理解原理"。

### 心态转变对照

| 传统心态 | AI 原生心态 |
|----------|-------------|
| 我要记住所有 API | 我要理解原理，AI 补细节 |

## 6.10 关键认知

三个核心原则：

1. **意图比语法重要**：清晰描述目标，AI 补充实现细节
2. **审查比信任重要**：AI 生成必须审查，尤其是安全性
3. **上下文是关键**：提供充足上下文，输出才贴合项目

### 最佳实践总结

| 实践 | 具体做法 |
|------|----------|
| 结构化提示词 | 背景 + 目标 + 约束 + 期望 |
| 项目导航文件 | 编写 CLAUDE.md，描述项目结构 |
| 代码审查习惯 | 每次生成必审查：安全、正确、可维护 |
| 验证优先 | 运行测试、手动验证，不盲目信任 |
| 迭代协作 | 反馈修正，多轮迭代，逐步完善 |
| 安全意识 | 明确安全要求，审查注入、加密、权限 |

### 常见陷阱速查

| 陷阱 | 一句话应对 |
|------|-----------|
| 过度信任 | AI 生成的代码 = 别人写的代码，必须审查 |
| 上下文不足 | 先写 CLAUDE.md，再开始开发 |
| 意图不清 | 用模板结构化提示词 |
| 依赖幻觉 | 查文档验证 API 存在性 |
| 安全忽视 | 明确"需要安全处理"，审查敏感操作 |
